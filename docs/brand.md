# 뜸 — Brand & Design Guidelines

핸드드립 레시피 계산기 앱 **뜸**의 철학, 보이스, 비주얼 언어 정의.
어떤 감각으로 만들지를 담는 문서.

---

## 이름

**뜸** — 한국어 한 글자.

핸드드립의 첫 단계. 원두에 물을 머금게 하고 약 30~45초간 **기다리는** 정적인 시간. 동시에 "뜸을 들이다"라는 관용구 — 서두르지 않고 충분히 기다린다는 뜻.

이 앱이 무엇을 하려 하는지 이름 하나가 이미 말함: 숫자는 도구고, 본질은 기다림이다.

---

## 철학

> **숫자는 디딤돌입니다.**
> 답이 아니라 딛고 설 자리. 기준이 있기에 안심하고 벗어날 수 있고, 돌아올 곳이 있기에 멀리 갈 수 있습니다.

이 문장이 이 앱의 모든 결정의 준거점. 기능을 추가할지 말지, 카피를 어떻게 쓸지, 애니메이션 속도를 얼마로 할지 — 판단이 흐려지면 이 문장으로 돌아온다.

### 핵심 원칙 세 가지

**1. 기준은 있되 강요하지 않는다.**
앱은 정확한 숫자를 준다. 그러나 그 숫자는 지키라는 명령이 아니다. "이게 출발선"이라는 조용한 제안. 유저가 벗어나도 앱은 간섭하지 않는다.

**2. 앱은 유저의 시간을 방해하지 않는다.**
드립은 집중의 시간이다. 알림, 축하, 권유, 질문 — 모두 이 집중을 깬다. 앱은 필요할 때만 말하고, 말할 때는 짧게 말한다. 기본 상태는 침묵.

**3. 확장은 성취가 아니라 발견이다.**
고수에게 더 나은 커피를 뽑게 하려는 게 아니다. Hoffmann만 쓰던 사람이 Kasuya를 만나는 것 — 이건 경쟁이 아니라 **자기 세계가 넓어지는 기쁨**이다. 뱃지, 레벨, 기록 같은 성취 장치는 철학에 반한다.

## 보이스 & 톤

### 레퍼런스

조용한 골목에 오래 자리를 지킨 핸드드립 카페 사장님.
손님이 오면 인사하지만 달라붙지 않는다. 묻지 않으면 설명하지 않는다. 하지만 물으면 정확히 안다. 친절이 아니라 **단정함**이 기본.

### 원칙

- 존댓말. 거리감 있으나 냉정하지 않음.
- 감탄사 없음. 이모지 없음. "!" 없음.
- 응원하지 않음.
- 한 문장이면 한 문장.
- 지시가 아니라 앎의 공유. "~하세요" 보다 "~입니다" / "~면 됩니다".
- 유저를 이끌지 않음. 유저가 물으면 답함.

### Do / Don't

| 상황           | ❌                                                                      | ✅                                               |
| -------------- | ----------------------------------------------------------------------- | ------------------------------------------------ |
| 첫 방문        | "뜸에 오신 걸 환영합니다! ☕ 시작해볼까요?"                             | (인사 없음. 기본 레시피가 이미 화면에)           |
| 맛 변경        | "더 달콤한 맛으로 조정되었어요!"                                        | (조용히 숫자만 갱신)                             |
| 복사 성공      | "링크가 복사되었어요 🎉"                                                | "복사되었습니다."                                |
| 입력 범위 초과 | "잘못된 값이에요. 다시 시도해주세요."                                   | "1~6명까지."                                     |
| 메서드 설명    | "Kasuya 4:6은 2016 월드 챔피언 테츠 카스야가 개발한 혁신적인 메서드로…" | "처음 40%가 맛을, 나머지 60%가 강도를 정합니다." |
| 팁             | "💡 프로의 꿀팁: 물을 천천히 나선형으로 부으세요!"                      | "물은 가운데부터, 나선을 그리며."                |
| 공유 유도      | "친구에게도 공유해보세요!"                                              | (공유 버튼만 놓여있음. 권유 없음)                |
| 빈 결과        | "아직 저장된 레시피가 없어요. 지금 만들어보세요!"                       | "없습니다."                                      |

### 문장 길이 지침

- 설명 1문장 = 대개 **20자 이내** 지향
- 팁 1~2문장, 각 **30자 이내**
- 메서드 소개 **3문장 이내**
- 이 제한이 강제력을 가져야 함. 길어지면 뭔가 잘못됨.

## 비주얼 언어

### 방향

**따뜻한 모던** (warm modern).
차가운 미니멀도, 풀 레트로도 아님.

- 차가운 미니멀(순백 + 검정)은 "평가받는 느낌". 병원이나 은행 같아져서 여유의 철학이 깨짐.
- 풀 레트로는 금방 피곤해지고 "꾸민 티" 남. 계산기 기능이 장식에 묻힘.
- 중간 지점: 모던 sans-serif와 깔끔한 레이아웃을 뼈대로, 종이색 배경과 따뜻한 포인트 컬러가 살을 붙임.

### 컬러

**3색 원칙**. 이 이상 쓰지 않음.

| 역할        | 톤                               | 대략 범위             |
| ----------- | -------------------------------- | --------------------- |
| 배경        | 따뜻한 오프화이트 (종이/미색)    | `#FBF7EF` ~ `#F5EFE4` |
| 주색 (잉크) | 짙은 먹색 또는 에스프레소 브라운 | `#2A241E` ~ `#3A2F26` |
| 포인트      | 웜 오커 / 로스티드 오렌지        | `#B8843F` ~ `#A86832` |

**금지**:

- 순백(`#FFFFFF`), 순흑(`#000000`)
- 차가운 회색 (cool gray)
- 파스텔 (연핑크, 민트, 라벤더)
- 채도 높은 원색 (비비드 레드/블루/그린)
- 그라디언트 (기본 UI에서)

포인트 컬러는 **아끼기**. 페이지당 1~2곳 이하. 강조가 넘치면 강조가 아님.

### 타이포그래피

**숫자가 명확하게 읽혀야 한다**는 게 최우선 제약. 계산기니까.

**추천 조합**:

- 본문 / UI: Pretendard (한글) + Inter (Latin) — 모던 sans, 숫자 가독성 좋음
- 숫자 표시: **tabular figures** 필수 (`font-variant-numeric: tabular-nums`). 값 변경 시 레이아웃이 흔들리지 않게.
- 브랜드 / 일부 강조: JetBrains Mono 또는 IBM Plex Mono를 **한 끗만** 사용. 제목 전체를 mono나 serif로 채우지 않음.

**스케일** (구현 기준 — 실제 토큰: `docs/design-tokens.md` § Typography scale):

| Role              | 클래스          | 값    | 용도                                     |
| ----------------- | --------------- | ----- | ---------------------------------------- |
| Meta              | `text-2xs`      | 10px  | 유닛·인디케이터·아주 작은 캡션           |
| Label (narrow)    | `text-xs`       | 12px  | 좁은 라벨·세그먼트                       |
| Label / Body      | `text-sm`       | 14px  | 표준 라벨·본문·버튼 내부                 |
| Body (emphasis)   | `text-md`       | 16px  | 강조 본문·다이얼로그 타이틀 보조         |
| Title             | `text-lg`       | 20px  | 섹션 헤더                                |
| Metric (small)    | `text-xl`       | 22px  | 시점·다음 푸어 프리뷰                    |
| Metric (medium)   | `text-2xl`      | 24px  | 경과 타이머·Wall 타이틀                  |
| Hero (complete)   | `text-hero-sm`  | 72px  | Complete 화면 총 시간                    |
| Hero (timer)      | `text-hero-lg`  | 96px  | Brewing 화면 저울 목표                   |

- `tabular-nums` 필수 (값 변경 시 레이아웃 흔들림 방지)
- 글꼴: Pretendard Variable → Inter → system sans

**금지**:

- serif 전면 사용 (레트로 과해짐)
- 손글씨 폰트
- 극단적 weight 대비 (black vs thin 같은 건 지양)

### 여백

압도적으로 많이.

- 한 화면에 보이는 1차 정보 **3~5개**까지로 제한. 그 이상은 스크롤이나 상세 뷰로.
- 섹션 간 여백을 인색하게 두지 않음. 답답하면 여유의 철학이 깨짐.
- 계산기 영역과 결과 영역 사이는 특히 넉넉하게.

기준: 모바일 한 화면(375×667 기준)에서 동시에 보이는 요소 수 세어보기. 6개 넘으면 재검토.

### 움직임

**느리다**.

- 기본 트랜지션: **320ms**. 일반 앱의 220ms보다 한 템포 늦게.
- 덜 중요한 전환: 400~500ms.
- Easing: `ease-out` 기본. spring, bounce 사용 안 함.
- 값 변경: **크로스페이드** (`opacity` 전환). 숫자가 튀지 않고 녹아들게.
- 진입/퇴장: `translateY` 4~8px 정도의 아주 작은 움직임 + opacity.

**금지**:

- 튀는 애니메이션 (bounce, elastic, overshoot)
- 로딩 스피너 (순간적 크로스페이드로 대체)
- 성공 체크마크 애니메이션
- 파티클, 색종이, 반짝임

### 질감과 디테일

- **모서리**: `border-radius: 8~12px`. 과도하게 둥글게 하지 않음 (현대 앱 트렌드의 과한 radius는 장난감 느낌).
- **그림자**: 일반 UI 요소(버튼, 카드, 입력, 세그먼트)는 그림자를 쓰지 않음. 경계는 배경 톤 차이(`bg-surface-inset` 등)로. 부상 레이어(세그먼트 선택 pill, popover, dialog)에 한해 매우 옅은 그림자를 허용하며 `shadow-raised` / `shadow-popover` / `shadow-dialog` semantic 토큰만 사용.
- **선**: 얇게 (`1px`), 먹색을 살짝 옅게 한 것(`opacity: 0.1~0.15`).
- **종이 질감**: 선택적. 도입한다면 **거의 보이지 않을 만큼 옅게**. 배경 이미지로는 쓰지 않고, 아주 낮은 opacity의 노이즈 레이어 정도.

### 아이콘과 일러스트

- 아이콘: **얇은 라인** (1.5px stroke). 채워진 아이콘 지양.
- 주제: 드리퍼, 주전자, 저울, 물방울, 원두. 의인화(얼굴, 감정) 없음.
- 일러스트가 필요하다면 **선 드로잉 한 톤** (잉크 색만). 채색 일러스트는 톤을 깸.
- 이모지 전면 금지 (UI 안에서).

## UX 원칙

### 1. 0-탭 기본값

> 앱을 열면 레시피가 이미 거기 있다.

- 첫 로딩 시 기본값: **V60 / Kasuya 4:6 / 1인분(15g) / medium roast / balanced / medium strength**.
- 제출 버튼 없음. "계산하기" 같은 CTA 없음.
- 유저가 아무것도 안 해도 유효한 레시피 한 잔이 완성되어 있음.

### 2. 조정, 설정이 아님

"설정"이라는 단어 쓰지 않음. 양식 냄새.

- 토글을 건드리면 결과가 즉시 반응.
- 유저는 **양식을 채우는 게 아니라 이미 있는 것을 다듬는** 것.

### 3. 강제하지 않음

- 온보딩 없음. 코치마크 없음. 투어 없음.
- 메서드 설명, 팁, 원두 정보 — 전부 탭해야 나옴.
- 첫 화면에 정보를 밀어넣지 않음. 궁금해지는 만큼만 파고들게.

### 4. 회복 가능성

- 모든 조작은 즉시 되돌릴 수 있음.
- "초기화" 버튼은 있되 숨어 있음 (상세 메뉴 안에).
- 파괴적 확인 다이얼로그 없음. 되돌릴 수 있으니 망설일 이유 없음.

### 5. 침묵을 기본값으로

- 토스트 메시지 최소화. 꼭 필요할 때(예: 링크 복사)만 3초 이내.
- 푸시 알림 없음.
- 빈 상태는 설명하려 들지 않음.

## 절대 하지 않는 것

구현 중에 자연스럽게 튀어나오기 쉬운 패턴들. 명시적으로 금지.

- 이모지 (☕, 🎉, ✨ 포함)
- 감탄부호 `!`
- "축하해요", "잘하셨어요", "완벽해요"
- "환영합니다", "시작해볼까요"
- 스플래시 스크린 (로고 풀스크린 애니메이션)
- 온보딩 투어, 코치마크, 툴팁 팝업
- 게이미피케이션 (뱃지, 스트릭, 레벨, 포인트)
- 푸시 알림
- "친구에게 공유하세요!" 류 권유
- "아직 가입 안 하셨나요?" 류 다크 패턴
- 로딩 스피너
- 축하 애니메이션 (색종이, 체크마크 애니, 파티클)
- 얼굴이나 감정이 있는 일러스트
- 그라디언트 버튼
- 네온/글로우 효과

## 카피 라이브러리

실제 UI에 쓸 문구. Claude Code 구현 시 이 문구를 우선 사용하고, 필요한 게 없으면 위 톤 규칙에 맞게 생성.

### 시스템 / 상태

- 링크 복사 성공: `복사되었습니다.`
- 복원 실패: `불러오지 못했습니다.`
- 입력 범위 초과 (인원수): `1~6명까지.`
- 입력 범위 초과 (커피 그램수): `5~50g까지.`

### 레이블

- 커피 그램수 입력 모드: `커피`
- 인원수 입력 모드: `인원`
- 드리퍼: `드리퍼`
- 메서드: `방식`
- 로스팅: `로스팅` (`라이트` / `미디엄` / `다크`)
- 맛: `맛` (`달게` / `균형` / `산뜻하게`)
- 강도: `강도` (`연하게` / `보통` / `진하게`)
- 총 물량: `물`
- 온도: `온도`
- 분쇄도: `분쇄`
- 총 시간: `시간`
- 푸어 스케줄: `차수`

### 메서드 설명 (한 화면 안에서)

**Kasuya 4:6**

> 처음 40%가 맛을, 나머지 60%가 강도를 정합니다.

**Hoffmann V60**

> 분쇄도로 맛을 조정합니다. 비율과 시간은 그대로.

**Kalita 펄스**

> 평평한 바닥이라 나누어 붓습니다. 물이 쌓이지 않도록.

### 메서드별 팁 (레시피 하단에 1~2줄)

**Kasuya 4:6**

> 주전자를 중심부터 나선을 그리며.
> 총 시간 3분 30초 ± 15초.

**Hoffmann V60**

> 뜸 직후와 마지막 푸어 뒤 가볍게 스월.
> 맛이 흐리면 분쇄를 살짝 가늘게.

**Kalita 펄스**

> 중심만 좁게. 외곽은 건드리지 않기.
> 수위가 일정하도록.

### 분쇄 힌트 (시각 비유)

- fine: `설탕 정도`
- medium-fine: `고운 소금 정도`
- medium: `굵은 소금 정도`
- medium-coarse: `거친 설탕 정도`
- coarse: `굵은 후추 정도`

### 빈 상태

- 최근 레시피 없음: `없습니다.` (또는 아예 표시 안 함)

## 디자인 체크리스트 (Claude Code 전달용)

구현 완료 전 각 화면에 대해 이 질문들로 점검:

**철학**

- [ ] 앱을 열자마자 레시피가 화면에 있는가?
- [ ] 제출/계산 버튼이 없는가?
- [ ] 유저가 아무것도 안 해도 완성된 결과가 보이는가?

**보이스**

- [ ] 감탄부호(`!`)가 있는가? → 제거
- [ ] 이모지가 있는가? → 제거
- [ ] "환영", "축하", "시작해볼까요" 류 카피가 있는가? → 제거
- [ ] 한 문장이 20자를 크게 넘는가? → 재작성 검토

**비주얼**

- [ ] 순백(`#FFF`)이나 순흑(`#000`)을 쓰고 있는가? → 오프화이트/먹색으로
- [ ] 한 화면에 3색 이상 포인트 컬러가 있는가? → 1색으로 줄이기
- [ ] 그라디언트, 네온, 글로우가 있는가? → 제거
- [ ] 아이콘이 채워진 형태인가? → 얇은 라인으로
- [ ] 숫자가 `tabular-nums`로 설정되어 있는가?

**움직임**

- [ ] 트랜지션이 220ms 이하로 빠른가? → 320ms로 늦추기
- [ ] 값 변경 시 숫자가 스냅되는가? → 크로스페이드로
- [ ] bounce, spring, elastic을 쓰는가? → ease-out으로

**UX**

- [ ] 강제 온보딩이 있는가? → 제거
- [ ] 푸시/토스트가 많은가? → 꼭 필요한 것만
- [ ] 확인 다이얼로그가 있는가? → 되돌릴 수 있게 만들어서 제거
